# 功能测试用例规范 v1.0

## 一、用例编写原则

1. **唯一性**：每条用例有唯一编号，格式为 `TC-[模块缩写]-[序号]`
2. **原子性**：每条用例只验证一个功能点或场景
3. **可重复性**：任何人按照步骤执行都能得到相同结果
4. **可追溯性**：用例必须关联到需求文档
5. **独立性**：用例之间不存在执行顺序依赖

## 二、用例分级标准

| 优先级 | 定义 | 使用场景 |
|--------|------|----------|
| P0 | 冒烟级 | 核心业务主流程，阻断性功能。如：登录、支付、核心数据提交 |
| P1 | 高优先级 | 重要功能的正常流程和关键异常流程。如：表单校验、权限控制 |
| P2 | 中优先级 | 次要功能和边界条件。如：分页、排序、筛选组合 |
| P3 | 低优先级 | UI细节、文案、提示信息、极端边界场景 |

## 三、测试点建模（新增）
在生成测试用例前，必须先抽取标准测试点；测试用例必须一一映射到测试点，不允许跳过 P0 测试点。每个测试点必须包含：

- 测试点编号（TO-XXX）
- 功能模块
- 场景描述
- 触发条件
- 预期行为
- 优先级（P0/P1/P2）

### 测试点分类（必须覆盖）
1. 功能流程（主流程 / 分支）
2. 状态流转（如：未登录 → 登录 → 返回）
3. 权限控制（会员 / 非会员 / 团队）
4. 配置控制（AB实验 / 开关）
5. 异常场景（网络 / 接口失败）
6. 边界条件（临界值 / 极值）

⚠️ 要求：
- 每个业务规则必须至少对应1个测试点
- 每个状态组合必须至少1个测试点


## 四、用例结构模板

每条功能测试用例必须包含以下字段：

```
用例编号: TC-XXX-001
用例标题: [模块名] - [具体验证点描述]
优先级: P0 / P1 / P2 / P3
所属模块: 一级模块 > 二级模块
前置条件: 执行该用例前必须满足的状态或数据条件
测试步骤:
  | 步骤序号 | 操作描述 | 输入数据 | 预期结果 |
  |---------|---------|---------|---------|
  | 1       |         |         |         |
预期结果: 最终预期的业务结果描述
```
## 五、状态维度覆盖（新增）

涉及状态变化的需求，必须覆盖用户状态、系统状态、环境状态、数据状态四类维度。
生成测试用例时，必须显式覆盖以下状态维度：

### 1. 用户状态
- 登录 / 未登录
- 会员 / 非会员
- 团队用户 / 个人用户

### 2. 系统状态
- AB实验命中 / 未命中
- 功能开关开启 / 关闭

### 3. 环境状态
- 网络正常 / 异常
- 接口成功 / 失败

⚠️ 要求：
- 每个状态至少1条用例
- 关键路径需组合状态（如：未登录 + AB命中）

## 六、生成约束（新增）

生成测试用例时必须满足：

1. 每个测试点至少生成1条用例
2. 不允许跳过任何P0测试点
3. 必须覆盖：
   - 所有业务分支
   - 所有状态组合
   - 所有异常路径
4. 若存在多条件组合（判定表），必须覆盖所有组合
5. 若发现测试点缺失，必须先补充测试点再生成用例
6.若需求包含 AB 实验、功能开关、策略配置、初始化参数、推送协议、定时任务等系统控制逻辑，必须单独设计测试用例覆盖其生效、不生效、缺失、异常四类场景。

## 七、用例设计方法

生成用例时，应综合使用以下测试设计方法：

### 4.1 等价类划分
- 将输入数据划分为有效等价类和无效等价类
- 每个等价类至少覆盖一条用例
- 例：手机号输入 → 有效(11位数字)、无效(10位/12位/含字母/空)

### 4.2 边界值分析
- 关注输入范围的边界点：最小值、最小值-1、最小值+1、最大值、最大值-1、最大值+1
- 例：密码长度6-20位 → 测试5位、6位、7位、19位、20位、21位

### 4.3 判定表法
- 当多个条件组合影响结果时，列出所有条件组合
- 例：优惠券使用条件（会员等级 × 订单金额 × 优惠券类型）

### 4.4 场景法
- 覆盖正常业务流程（基本流）
- 覆盖每个可能的分支和异常（备选流）
- 例：下单流程 → 正常下单、库存不足、地址无效、支付超时

### 4.5 错误推测法
- 基于经验推测可能出错的场景
- 包括：空值、特殊字符、超长输入、并发操作、网络异常

## 八、覆盖要求

每个功能模块的用例必须覆盖以下维度：

1. **正向流程**：至少1条主流程 P0 用例
2. **参数校验**：每个输入字段的有效/无效边界
3. **业务规则**：所有业务判断分支
4. **异常处理**：网络异常、数据异常、权限异常
5. **状态流转**：涉及状态变化时，覆盖所有状态转换路径
6. **数据完整性**：创建/修改/删除后数据一致性验证

## 九、用例输出格式

生成的每条用例使用以下 Markdown 格式输出：

```markdown
## TC-XXX-001 [模块] - 用例标题

- **优先级**: P1
- **所属模块**: 一级模块 > 二级模块
- **前置条件**: 用户已登录且处于首页

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 点击"XX"按钮 | 弹出XX弹窗 |
| 2 | 输入XXX | 输入框显示XXX |
| 3 | 点击"确认" | 提示"操作成功"，页面跳转至XX |

- **预期结果**: 数据保存成功，列表中可见新记录
```

## 十、注意事项

1. 步骤描述使用祈使句，明确具体操作对象和动作
2. 预期结果必须可验证，避免模糊描述如"正常显示"
3. 涉及数据的用例，明确写出测试数据示例
4. 前置条件要完整，包括账号状态、数据准备、页面位置等
5. 一条用例的测试步骤建议不超过10步，超过时考虑拆分
