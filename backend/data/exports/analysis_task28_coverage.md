# 任务ID28 用例生成覆盖率分析报告

## 一、现状概述

任务ID28 针对「用户日常唤活送会员策略（安卓单端）」需求，使用 doubao-pro-32k 模型生成了 13 条功能测试用例，经分析脚本评估覆盖率仅为 21.7%。

对比同一需求的任务ID27（使用增强模式 Prompt），ID27 提取了 28 个标准测试点并生成了 10 条测试用例且每条用例显式映射到测试点，结构化程度和覆盖意识明显更优。

## 二、根因分析（共识别 4 个层面问题）

### 根因1：需求文档中 56.3% 的内容是 SQL 代码（严重）

需求文档总长 6383 字符，其中 SQL 代码占 3596 字符（56.3%），这些数据工程查询对功能测试用例设计毫无帮助，却严重占用了模型的有效上下文窗口。模型需要在大量无关 SQL 代码中「寻找」真正的业务规则，导致：

- 对人群规则的理解不够精细（高价值用户、稳定低频用户的量化定义被 SQL 实现淹没）
- 对业务逻辑的边界条件挖掘不够深入

此外，需求文档还存在以下质量问题：
- `[图片]` 占位符（钉钉文档中的图片未被提取，需求背景信息缺失）
- 「需求划分」表格为空
- 「怎么AB？」这样的未确认问题留在文档中
- 已删除线内容（`~~在用户上次启动app时间发送~~`）仍保留在原文中

### 根因2：Prompt 模板退化——移除了对复杂需求至关重要的「增强模式」（严重）

对比两次生成的 Prompt：

| 维度 | 任务27（增强模式 Prompt） | 任务28（当前 Prompt） |
|------|--------------------------|----------------------|
| 需求复杂度判断 | 有（自动识别大需求/小需求） | 无 |
| 模块拆分 Step0 | 有（按人群/入口/主流程/状态/配置/异常等拆分） | 无 |
| 测试点输出要求 | **必须输出**测试点清单（作为第一部分） | 「内部执行，不直接输出中间结果」 |
| 用例→测试点映射 | 必须标注 TO-XXX 映射 | 无映射要求 |
| 策略类需求约束 | 「策略/人群/规则类必须独立成测试点，不允许用UI测试点替代」 | 无此约束 |

任务27 的 Prompt 明确要求 AI 先输出测试点清单再生成用例，并且要求每条用例标注对应的测试点编号（TO-XXX），这种强制的两步结构化过程显著提升了覆盖率。

而任务28 的当前 Prompt 让 AI 在「内部」完成测试点抽取，不要求输出，这导致 AI 倾向于跳过或简化这一关键步骤，直接生成笼统的测试用例。

### 根因3：Pipeline 未对需求文档做预处理（中等）

`functional_pipeline.py` 直接将原始需求文档全文塞入 Prompt，未做任何预处理：
- 未过滤 SQL 代码块
- 未提取/结构化关键业务规则
- 未对不完整内容（空表格、占位符）做标记或警告
- `target_module` 和 `expected_count` 参数始终为空字符串，未被利用

### 根因4：缺少生成后的覆盖率自动验证（中等）

Pipeline 在生成完成后仅做了「是否解析出用例」的校验，没有对覆盖率进行自动评估：
- 未检查生成的用例数量是否合理
- 未检查是否覆盖了需求中的所有关键模块
- 未验证 P0 测试点是否 100% 覆盖

## 四、优化方案

### 方案1：恢复并增强 Prompt 模板（优先级最高，预期提升覆盖率 30-40pp）

将任务27 的「增强模式」Prompt 中的关键结构恢复到当前模板中，具体改动：

1. 恢复需求复杂度判断（大需求自动进入增强模式）
2. 恢复模块拆分 Step0
3. **强制要求输出测试点清单**（不再是「内部执行」）
4. 强制要求每条用例标注对应测试点编号
5. 增加策略/人群/规则类需求的专项约束

### 方案2：Pipeline 增加需求预处理（预期提升 10-15pp）

在 `functional_pipeline.py` 的 `_generate` 方法中增加预处理步骤：

```python
# 在渲染 Prompt 之前增加预处理
requirement_content = self._preprocess_requirement(requirement_content)
```

预处理逻辑：
- 移除 SQL/代码块（保留代码块前后的业务描述）
- 将 `[图片]` 标记为「⚠️ 此处有图片未提取，可能包含关键信息」
- 移除空表格
- 移除删除线内容

### 方案3：两阶段生成（预期提升 15-20pp）

将当前的单次 AI 调用改为两阶段：

- 阶段1：专门调用 AI 提取结构化测试点（JSON 格式），并自动校验完整性
- 阶段2：基于确认后的测试点列表生成具体测试用例

这样即使第二阶段有遗漏，也能通过测试点列表发现。

### 方案4：生成后自动覆盖率校验（预期提升 5-10pp）

在 Pipeline 中增加自检步骤：
- 统计生成用例覆盖的模块数 vs 需求中涉及的模块数
- 检查 P0 用例数量是否合理（至少占总用例的 30%）
- 若覆盖率评估不达标，自动触发补充生成

## 五、总结

覆盖率仅为 21.7% 的核心原因是：当前 Prompt 模板在迭代中退化（移除了增强模式），叠加需求文档中 56% 的无关 SQL 代码噪声，导致 AI 模型对业务规则的挖掘深度严重不足。

建议优先实施方案1（恢复增强 Prompt）和方案2（需求预处理），预期可将覆盖率从 21.7% 提升至 60-75%。再叠加方案3和方案4，目标可达 80%+。
